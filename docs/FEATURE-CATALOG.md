# Feature catalogue and SDK audit

Audited against the source tree on 2026-07-13. The manifest version at the
time of the audit is `1.4.4`.

The audit uses the following evidence, in descending order of confidence:

1. runtime wiring in [`src/extension.ts`](../src/extension.ts);
2. providers, managers, views and the transactional store under [`src/`](../src/);
3. automated tests under [`src/test`](../src/test),
   [`src/transactional/__tests__`](../src/transactional/__tests__) and
   [`src/anchoring/__tests__`](../src/anchoring/__tests__);
4. contributions in [`package.json`](../package.json);
5. release history in [`CHANGELOG.md`](../CHANGELOG.md).

Documentation or a manifest contribution alone is not treated as proof that a
workflow is operational.

## Status legend

| Status      | Meaning                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------ |
| **Stable**  | Wired into the active extension and covered by an automated test or a directly verifiable implementation contract. |
| **Partial** | Usable code exists, but wiring, parity, safety, accessibility or automated validation is incomplete.               |
| **Planned** | No complete runtime capability exists; the work is recorded in [`tasks.jsonp`](../tasks.jsonp).                    |
| **Blocked** | The requested interaction cannot be implemented with the current stable Microsoft VS Code API.                     |

“Validation” below names existing tests; it does not assert that an unexecuted
working tree is green. “Introduced” is taken from the changelog when it is
knowable. Task links open the ledger; search for the displayed task ID.

The source-comment syntax catalogue contains 42 IDs in total: 37 primary VS
Code language modes covered by the required catalogue test plus five
compatibility aliases/extras (`shell`, `bash`, `zsh`, `clojurescript`,
`scheme`). Scanner preview coverage spans all 42 entries. Reverse insertion is
restricted to TypeScript, JavaScript, C, C++, Java, C#, Go, Rust, Kotlin, Dart,
Python, TOML, Lua, CSS, SCSS and Clojure, followed by a position-level lexical
safety check. Destructive source-comment removal is a separate, narrower
allowlist: only line syntax in TypeScript, JavaScript, Java, C, C++, C#, Go,
Rust, Kotlin and Dart. Block/docblock records and all other scanner modes remain
Keep-only. The remaining modes and block-removal contexts require
parser/source-aware placement or replacement.

Current generated markers retain the readable eight-character ID fragment and
add a 128-bit SHA-256 fingerprint; legacy short markers remain readable but do
not satisfy the strong proof required by a destructive move. Before an
annotation is removed, its encoded comment and the saved source scan must both
recover the exact full identity and message. Deduplication reserves exact
provenance, then strong identity, then legacy fallbacks bijectively, so one
annotation never suppresses two equal same-line comments. If source rollback
is rejected or throws, the destination is conservatively retained and strict
persistence is attempted. A cascading failure is surfaced rather than
reporting success with zero representations.
